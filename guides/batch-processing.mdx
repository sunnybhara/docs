Process large volumes of text efficiently while respecting rate limits and minimizing costs.

## Strategy overview

| Technique | Benefit |
|-----------|---------|
| **Concurrency control** | Stay within rate limits |
| **Semantic caching** | Avoid paying for duplicate/similar texts |
| **Exponential backoff** | Gracefully handle 429 errors |
| **Progress tracking** | Resume interrupted batches |

## TypeScript batch processing

```typescript
import { Nawa } from '@nawalabs/sdk'

const nawa = new Nawa({
  apiKey: process.env.NAWA_API_KEY,
  maxRetries: 3
})

interface Comment {
  id: string
  text: string
}

async function processBatch(comments: Comment[], concurrency = 5) {
  const results = []
  const queue = [...comments]

  // Process with controlled concurrency
  const workers = Array.from({ length: concurrency }, async () => {
    while (queue.length > 0) {
      const comment = queue.shift()!

      const { data, error } = await nawa.classify({
        text: comment.text
      })

      if (error?.type === 'rate_limit_error') {
        // Put it back and wait
        queue.unshift(comment)
        const delay = error.retryAfter ?? 2000
        await new Promise(r => setTimeout(r, delay))
        continue
      }

      results.push({
        id: comment.id,
        classification: data,
        error: error?.message
      })
    }
  })

  await Promise.all(workers)
  return results
}

// Usage
const comments = [
  { id: '1', text: 'متى الجزء الثاني؟' },
  { id: '2', text: 'ما شاء الله عليك' },
  { id: '3', text: 'الصوت وحش أوي' },
  // ... thousands more
]

const results = await processBatch(comments, 5)
console.log(`Processed ${results.length} comments`)
```

## Python batch processing

```python
import asyncio
from nawa import AsyncNawa

async def process_batch(comments: list[dict], concurrency: int = 5):
    async with AsyncNawa(api_key="nawa_live_sk_xxx") as nawa:
        semaphore = asyncio.Semaphore(concurrency)
        results = []

        async def classify_one(comment):
            async with semaphore:
                result = await nawa.classify(
                    text=comment["text"]
                )
                return {
                    "id": comment["id"],
                    "classification": result.data,
                    "error": str(result.error) if result.error else None
                }

        tasks = [classify_one(c) for c in comments]
        results = await asyncio.gather(*tasks)
        return results

# Usage
comments = [
    {"id": "1", "text": "متى الجزء الثاني؟"},
    {"id": "2", "text": "ما شاء الله عليك"},
    {"id": "3", "text": "الصوت وحش أوي"},
    # ... thousands more
]

results = asyncio.run(process_batch(comments, concurrency=5))
print(f"Processed {len(results)} comments")
```

## Cost optimization tips

<CardGroup cols={2}>
  <Card title="Leverage semantic cache" icon="bolt">
    Similar texts produce cache hits (`X-NAWA-Cache: HIT`), which are free. Typical cache hit rates are 20-30% for comment workloads.
  </Card>
  <Card title="Deduplicate before sending" icon="filter">
    Remove exact duplicates from your batch before sending. This saves both credits and latency.
  </Card>
  <Card title="Use rubric classify when possible" icon="list-check">
    At $0.003 per call (vs $0.006 for classify), `/v1/rubric-classify` costs half when you need custom categories.
  </Card>
  <Card title="Monitor with /v1/usage" icon="chart-line">
    Check your cache hit rate and per-endpoint costs to identify optimization opportunities.
  </Card>
</CardGroup>

## Rate limit handling

Your batch processor should respect rate limits:

| Tier | Requests/min | Recommended concurrency |
|------|-------------|------------------------|
| Free | 10 | 1-2 |
| Trial | 60 | 5-10 |
| Growth | 120 | 10-15 |
| Enterprise | 300 | 20-30 |
| Enterprise+ | 1,000 | 50-100 |

<Tip>
  Semantic cache hits do not count toward rate limits. Higher cache hit rates effectively increase your throughput beyond the tier limit.
</Tip>

<Warning>
  Do not retry in a tight loop without backoff. This will extend your rate limit window and may result in longer delays.
</Warning>
